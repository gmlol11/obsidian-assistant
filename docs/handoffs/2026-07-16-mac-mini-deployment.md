# Handoff: развёртывание Telegram/OpenClaw на Mac mini

Дата фиксации: **2026-07-16**, часовой пояс `Europe/Moscow`.

Этот документ — точка входа для новой рабочей сессии. Он фиксирует состояние репозитория и порядок первого развёртывания. Канонические требования остаются в [руководстве по Mac mini](../deployment-macos.md), [runbook Telegram/OpenClaw](../openclaw-telegram.md), [модели угроз](../threat-model.md) и [ADR-0007](../decisions/0007-use-openclaw-native-telegram-and-typed-capture-command.md).

## Цель следующей сессии

Безопасно развернуть уже реализованный текстовый поток:

```text
Telegram /capture
  -> OpenClaw long polling
  -> owner-only capture plugin
  -> строгий stdin bridge
  -> локальная очередь
  -> Vault Worker
  -> create-only Markdown в разрешённой папке
```

LLM, обычный диалог с ассистентом, голос, файлы, классификация, напоминания и ежедневный обзор **не входят** в это развёртывание.

## Зафиксированное состояние

- Репозиторий: `gmlol11/obsidian-assistant`.
- Целевая ветка: `main`.
- Рабочая ветка этапа: `agent/telegram-openclaw`.
- Реализация начинается с commit `c04c30b` (`feat: add secure Telegram OpenClaw capture bridge`).
- Draft PR: [#3 — secure Telegram and OpenClaw capture bridge](https://github.com/gmlol11/obsidian-assistant/pull/3).
- PR создан поверх merge commit этапа Capture `2841e23`.
- Последняя полная локальная проверка: 40 Python-тестов и 6 тестов плагина.
- GitHub CI для commit `c04c30b` прошёл: Python 3.12, Python 3.13, OpenClaw plugin и container — успешно.
- Плагин проверен с OpenClaw `2026.7.1` и Node.js `24.15.0`.
- Deep security audit в изолированной проверочной конфигурации: 0 critical; предупреждения относились к незапущенному Gateway и отсутствующему reverse proxy при loopback.

PR остаётся черновиком. Перед развёртыванием новой сессии нужно проверить его актуальный head и получить явное решение владельца о merge. Предпочтительный источник развёртывания — `main` после merge, а не изменяемая рабочая ветка.

## Что ещё не сделано

- не создан или не подключён реальный Telegram-бот;
- реальные bot token, Gateway token и numeric owner ID нигде не сохранены в репозитории;
- OpenClaw Gateway не запускался на целевом Mac mini;
- не проведён read-only аудит целевого Mac mini;
- не выбраны окончательные локальные пути, сервисный пользователь и способ автозапуска;
- реальный vault не подключался к worker;
- не выбран Desktop Sync, Headless Sync или работа без Obsidian Sync;
- не проверены backup/restore и восстановление после перезагрузки.

## Неизменяемые границы

1. Не просить пользователя присылать токены, ключи или заполненные env-файлы в чат.
2. Не выводить секреты командами диагностики, в diff, журналы или GitHub.
3. Начинать с `OBSIDIAN_DRY_RUN=true` и тестового vault.
4. Не направлять тесты на реальный vault.
5. Не переключать реальный vault в apply-режим без отдельного явного подтверждения пользователя.
6. Разрешать запись только в `00 Inbox`; базовая операция — создание нового файла без перезаписи.
7. Gateway оставлять на loopback; не открывать порт на роутере, не добавлять публичный reverse proxy и не включать webhook Telegram.
8. OpenClaw должен оставаться единственным потребителем Telegram `getUpdates` для этого bot token.
9. Worker env не должен наследовать Telegram, Gateway или LLM credentials.
10. Не помещать runtime внутрь vault: pending и quarantine могут содержать полный входящий текст.
11. Не давать OpenClaw произвольный shell или прямой доступ ко всему vault.
12. Не переносить в систему конфиденциальную информацию MOEX.

## Что потребуется от владельца

В начале новой сессии подтвердить без передачи секретов:

- появился ли прямой терминальный доступ к Mac mini;
- можно ли выполнять административную подготовку отдельного сервисного пользователя;
- создан ли отдельный Telegram-бот через официальный `@BotFather`;
- известен ли владельцу собственный numeric Telegram user ID;
- где локально будет тестовый vault и где расположен реальный vault;
- используется ли на Mac mini Obsidian Desktop и какой механизм уже синхронизирует vault;
- готова ли проверяемая резервная копия реального vault.

Bot token, Gateway token и numeric owner ID владелец вводит только локально на Mac mini. В публичных отчётах owner ID также лучше маскировать.

## Порядок развёртывания

### 0. Сверить исходную точку

1. Прочитать `AGENTS.md`, этот handoff и связанные канонические документы.
2. Проверить чистоту рабочей копии и текущую ветку.
3. Проверить состояние draft PR #3, его актуальный head, diff и CI.
4. Не смешивать найденные пользовательские изменения с развёртыванием.
5. После явного согласия владельца завершить review/merge PR #3.
6. Зафиксировать точный commit или release, который будет развёрнут.

### 1. Провести read-only аудит Mac mini

До установки или изменения настроек собрать локально и сообщить только безопасное резюме:

- модель процессора/архитектуру, объём памяти и свободное место;
- версию macOS, состояние обновлений и FileVault;
- текущего пользователя и наличие отдельного сервисного пользователя;
- версии Git, Python, Node.js, npm, Docker и OpenClaw, если они уже есть;
- существующие слушающие TCP-порты и занятость loopback-порта Gateway;
- режим сна, автозапуска после сбоя питания и необходимость ручного входа;
- существующие backup и sync-механизмы;
- уже работающие процессы OpenClaw, Telegram-боты или Vault Worker.

Не публиковать hostname, IP домашней сети, серийный номер, имена Wi-Fi, абсолютные домашние пути и полный список посторонних сервисов.

После аудита остановиться и согласовать план изменений, если обнаружены конфликтующие сервисы, неподдерживаемый runtime или конкурирующая синхронизация.

### 2. Подготовить изолированную среду

1. Создать или выбрать непривилегированного сервисного пользователя без доступа к основному Keychain и SSH-ключам.
2. Разделить каталоги репозитория, runtime, тестового vault и реального vault.
3. Настроить права так, чтобы сервис видел только необходимые каталоги.
4. Установить Python 3.12+ и поддерживаемую OpenClaw версию Node.js.
5. Для зафиксированного OpenClaw `2026.7.1` использовать Node.js `22.22.3+` в ветке 22 или `24.15.0+` в ветке 24.
6. Не обновлять зависимости сверх зафиксированных версий без отдельной проверки.

### 3. Установить проект и повторить проверки

1. Клонировать уже одобренный commit из `main`.
2. Создать Python venv и установить проект.
3. Выполнить `make check` до подключения реального vault.
4. Собрать tarball плагина командой из runbook.
5. Установить плагин штатным менеджером OpenClaw.
6. Проверить `plugins inspect --runtime --json`: статус `loaded`, команда `capture`, конфигурационная схема доступна.

Если проверка на Mac mini отличается от зелёного CI, не продолжать к секретам и vault до диагностики причины.

### 4. Создать локальную конфигурацию

Использовать шаблоны из `deploy/openclaw`:

- `openclaw.env.example` -> `~/.openclaw/.env`;
- `openclaw.example.json5` -> `~/.openclaw/openclaw.json`;
- `worker.env.example` -> отдельный `worker.env` рядом с сервисом.

Установить права `600`. Заполнить значения локально, не копируя их в чат. Пути к bridge и worker env должны быть абсолютными.

Проверить разделение полномочий:

- OpenClaw env: bot token, Gateway token, owner ID и пути;
- OpenClaw JSON: подстановки переменных, allowlist, loopback и policy;
- worker env: vault/runtime/allowlist/dry-run, без Telegram/Gateway/LLM-секретов.

На этом шаге worker должен указывать только на тестовый vault, а `OBSIDIAN_DRY_RUN` должен оставаться `true`.

### 5. Проверить конфигурацию до первого сообщения

1. Выполнить `doctor` для worker.
2. Выполнить OpenClaw config validation.
3. Запустить `openclaw doctor`.
4. Выполнить `openclaw security audit --deep` до и после запуска Gateway.
5. Убедиться, что Gateway слушает только loopback.
6. Убедиться, что Telegram использует long polling, а webhook и публичный ingress отсутствуют.
7. Проверить, что в логах нет токенов, owner ID и текста тестовой заметки.

Любая critical-находка security audit блокирует продолжение.

### 6. Выполнить Telegram smoke test в dry-run

1. Запустить Gateway вручную в контролируемой терминальной сессии.
2. Проверить `channels status --probe`.
3. Отправить владельцем в личный чат: `/capture Проверка dry-run`.
4. Ожидать metadata-only ответ с `🧪`, request ID и будущим относительным путём.
5. Подтвердить, что файл в vault не создан, а событие осталось `pending`.
6. Проверить queue status без вывода payload.

После smoke test сохранить только обезличенное резюме проверки, без текста сообщения и идентификаторов.

### 7. Проверить запреты

До разрешения записи подтвердить:

- другой Telegram account не может выполнить `/capture`;
- групповой чат не может выполнить команду;
- обычный текст не запускает capture-команду;
- повтор одного request ID не создаёт второй элемент или файл;
- Gateway недоступен не-loopback интерфейсам;
- плагин не может запустить произвольную команду;
- worker не видит Telegram/Gateway/LLM-секреты;
- попытка пути вне allowlist отклоняется;
- существующий файл не перезаписывается.

Если второго Telegram account нет, отметить негативный тест как незавершённый, а не считать его пройденным по предположению.

### 8. Проверить ограниченную запись

1. Сначала переключить apply только для отдельного тестового vault.
2. Отправить новое уникальное `/capture` сообщение.
3. Подтвердить создание ровно одного Markdown-файла в тестовом `00 Inbox`.
4. Повторно проверить идемпотентность и запрет перезаписи.
5. Вернуть `dry-run=true` до подключения реального vault.
6. Подготовить и проверить резервную копию реального vault.
7. Подключить реальный vault сначала только в dry-run и просмотреть будущий путь.
8. Запросить у владельца отдельное явное подтверждение перед `OBSIDIAN_DRY_RUN=false` для реального vault.
9. После подтверждения выполнить один контролируемый create-only тест в `00 Inbox` и снова проверить backup.

### 9. Настроить эксплуатацию

Только после успешных ручных тестов:

- установить Gateway/worker в автозапуск с минимальными правами;
- проверить перезапуск процесса после сбоя;
- проверить поведение после обычной перезагрузки Mac mini;
- настроить metadata-only журналы и ротацию;
- настроить backup runtime и vault раздельно;
- выполнить тестовое восстановление;
- сохранить процедуру остановки, ротации токена и отката;
- начать недельный параллельный период без двух writer для одного входящего события.

## Стоп-условия

Немедленно остановить развёртывание и не расширять права, если:

- рабочая копия содержит неизвестные или несвязанные изменения;
- PR #3 не прошёл проверки или развёртываемый commit не определён;
- обнаружен второй потребитель того же bot token, Telegram webhook или второй OpenClaw Gateway;
- Gateway слушает публичный интерфейс;
- security audit сообщает critical;
- worker env содержит сетевые секреты;
- runtime находится внутри vault;
- тест затрагивает реальный vault до явного разрешения;
- активны два sync-механизма для одного vault;
- отсутствует проверяемая резервная копия перед первой реальной записью;
- версия Node.js не поддерживается выбранной версией OpenClaw.

## Критерии завершения этого развёртывания

- PR #3 рассмотрен и развёрнут конкретный commit из `main`;
- Gateway работает на Mac mini только через loopback и переживает обычный перезапуск;
- Telegram принимает `/capture` только от numeric owner ID в личном чате;
- dry-run создаёт очередь и preview, но не файл;
- apply создаёт ровно один новый файл только в разрешённом `00 Inbox`;
- негативные проверки allowlist и границ либо пройдены, либо явно отмечены как блокирующие;
- deep security audit не содержит critical;
- секреты отсутствуют в Git, чатах, логах и worker env;
- backup и тест восстановления зафиксированы;
- существует понятный откат к остановленному Gateway и `dry-run=true`.

- развёрнутый commit и версии runtime;

- итоговый статус plugin inspect, doctor и security audit;
