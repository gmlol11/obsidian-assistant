# Runbook: реагирование на инцидент

## Признаки

- неизвестный отправитель получил ответ агента;
- появились файлы вне разрешённой папки;
- заметка была перезаписана или удалена;
- Gateway оказался доступен из внешней сети;
- токен или секрет попал в журнал, Git или сообщение;
- установлен непроверенный skill/plugin;
- резко выросло число запросов к модели или расходы.

## Немедленные действия

1. Остановить OpenClaw и Vault Worker.
2. Отозвать затронутые Telegram/API токены.
3. Не удалять логи, очередь и карантин до фиксации времени и фактов.
4. Отключить публичный сетевой доступ или tunnel.
5. Создать защищённую копию текущего состояния vault и конфигурации без публикации.
6. Если затронуты заметки, не запускать Sync до оценки повреждения.

## Анализ

- определить первый подозрительный request ID;
- проверить actor ID, операцию, путь и версию приложения;
- сравнить изменённые файлы с backup/version history;
- проверить OpenClaw config, plugins, skills и результаты security audit;
- проверить Git history и журналы на утечку секретов;
- установить, могла ли компрометация распространиться на другие сервисы Mac mini.

## Восстановление

1. Исправить причину и добавить воспроизводящий тест.
2. Заменить все потенциально раскрытые секреты.
3. Восстановить данные по backup runbook.
4. Запустить систему с новым окружением в dry-run.
5. Проверить allowlist, loopback Gateway и запрет shell.
6. Разрешить только capture в Inbox.
7. Наблюдать не менее одного полного цикла ежедневного обзора.

## После инцидента

- создать приватный отчёт с временной шкалой;
- обновить модель угроз, ADR и runbook;
- сообщить об уязвимости приватно, если она находится в стороннем проекте;
- не публиковать реальные заметки, токены и домашнюю инфраструктуру в issue.
