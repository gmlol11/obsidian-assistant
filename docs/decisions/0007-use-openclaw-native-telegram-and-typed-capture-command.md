# ADR-0007: Использовать нативный Telegram OpenClaw и типизированную команду Capture

- Статус: accepted
- Дата: 2026-07-16

## Контекст

Telegram Bot API не допускает независимое чтение `getUpdates` несколькими процессами без конкуренции за один поток обновлений. Одновременно Capture должен сохранять исходный текст при недоступности LLM, а сетевой агент не должен получать произвольный shell или прямой файловый доступ к vault.

## Решение

Использовать нативный Telegram-канал OpenClaw как единственного клиента Bot API. Long polling остаётся транспортом по умолчанию, поэтому входящие порты не открываются.

Установить отдельный плагин `obsidian-assistant-capture`, который регистрирует `/capture` как нативную команду OpenClaw:

- команда выполняется до обращения к LLM;
- OpenClaw проверяет channel allowlist, затем плагин повторно сверяет numeric sender ID со своим `ownerIds`;
- плагин запускает только один абсолютный `obsidian-assistant` executable без shell;
- версия события `capture.text` передаётся через stdin, а не через аргументы процесса;
- дочерний процесс получает очищенное окружение без Telegram, Gateway и LLM-секретов;
- отдельный worker env-файл определяет vault, allowlist путей и `dry-run`;
- ответ содержит только request ID, состояние очереди и относительный путь заметки.

При `processImmediately=true` bridge обрабатывает именно созданный запрос и сразу сообщает `completed`, `previewed`, `retry` или `quarantine`. Эта настройка не может отключить `dry-run`: режим записи контролируется только worker-конфигурацией.

## Последствия

- Capture не зависит от LLM и не расходует модельный запрос;
- один OpenClaw владеет Telegram long polling и его provider-level dedupe;
- посторонний sender ID должен пройти две независимые проверки;
- OpenClaw может инициировать только строго определённую операцию worker, а не произвольную команду;
- для реального запуска нужен совместимый с OpenClaw Node.js и отдельная установка плагина;
- неопределённый сбой между постановкой в очередь и Telegram-ответом требует проверки request ID, а не слепой повторной отправки другого текста.

## Рассмотренные варианты

- **Отдельный Python Telegram poller с тем же bot token:** отклонён из-за конкуренции двух `getUpdates` клиентов.
- **Agent tool или skill:** отклонён для базового Capture, потому что вызов зависел бы от модели и prompt-инструкций.
- **Прямой доступ OpenClaw к vault или shell:** отклонён из-за слишком широкой границы доверия.
- **Webhook:** не нужен для одного домашнего сервера и потребовал бы входящий HTTPS endpoint.
