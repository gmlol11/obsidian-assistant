# Capture и локальная очередь

## Назначение

Capture принимает текст, проверяет его до сохранения и помещает в долговечную очередь вне Obsidian vault. Очередь нужна, чтобы перезапуск процесса, временная ошибка диска или повторная доставка не приводили к тихой потере материала или неконтролируемому дубликату.

Текущая реализация рассчитана на одного владельца, один Mac и локальную файловую систему. Она не является распределённой очередью для нескольких серверов.

## Событие версии 1

Поддерживается только `event_type: capture.text`:

```json
{
  "schema_version": 1,
  "request_id": "12345678-1234-5678-1234-567812345678",
  "event_type": "capture.text",
  "source": "local",
  "actor_id": "local-owner",
  "created_at": "2026-07-16T09:30:00.000000Z",
  "payload": {
    "title": "Тестовая идея",
    "text": "Безопасный текст фикстуры"
  }
}
```

Неизвестные поля, другая версия, некорректный UUID, время без часового пояса, пустой текст и неподдерживаемый источник отклоняются до постановки в очередь. Fingerprint — SHA-256 канонического JSON всего события.

`request_id` — ключ идемпотентности:

- тот же ID и тот же fingerprint возвращают `DUPLICATE` без нового файла;
- тот же ID и другое содержимое возвращают конфликт;
- имя итоговой заметки детерминировано временем события и полным UUID.

## Хранение

Путь задаётся `OBSIDIAN_RUNTIME_PATH` и обязан находиться отдельно от vault:

```text
runtime/
└── queue/
    ├── pending/
    ├── processing/
    ├── completed/
    ├── quarantine/
    └── .lock
```

`pending`, `processing` и `quarantine` могут содержать исходный текст. Они исключены из Git, не выводятся командами статуса и должны быть доступны только владельцу сервиса. После успеха полное событие удаляется, а в `completed` остаётся metadata-only квитанция: request ID, fingerprint, источник, число попыток, время и относительный путь заметки.

## Обработка

1. `enqueue` атомарно записывает проверенное событие в `pending`.
2. Worker под файловой блокировкой перемещает его в `processing`, увеличивает число попыток и устанавливает lease.
3. Vault Writer создаёт новый Markdown-файл только в allowlist.
4. Успех создаёт квитанцию в `completed` и удаляет полное событие из `processing`.
5. Временная ошибка возвращает событие в `pending` до лимита попыток.
6. Постоянная ошибка или исчерпанный лимит перемещает событие в `quarantine`.

Если процесс завершился после записи заметки, но до квитанции, lease возвращает событие в `pending`. Повторная попытка считает успехом только существующий файл с полностью идентичными байтами. Отличающийся файл никогда не перезаписывается и приводит к карантину.

## Команды

```bash
obsidian-assistant --env-file .env queue enqueue \
  --title "Идея" --text "Обезличенный материал"
obsidian-assistant --env-file .env queue status
obsidian-assistant --env-file .env queue process
obsidian-assistant --env-file .env queue process --limit 10 --apply
obsidian-assistant --env-file .env queue recover
obsidian-assistant --env-file .env queue retry REQUEST_ID
```

При `OBSIDIAN_DRY_RUN=true` обычный `process` только показывает request ID и будущий относительный путь, не захватывает элемент и не пишет в vault. `--apply` является явным подтверждением записи.

## Ограничения

- `fcntl` и локальная семантика файловой системы предполагают macOS или Linux;
- одновременно разрешён один логический экземпляр очереди на одном runtime;
- runtime нельзя размещать на сетевом диске с неизвестной семантикой lock/rename;
- очередь пока принимает только текст; файлы, голос и Telegram появятся отдельными этапами;
- автоматическая очистка completed-квитанций пока не реализована.
