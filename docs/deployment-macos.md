# Развёртывание на Mac mini

Документ описывает целевой перенос после появления прямого доступа. Код и безопасные шаблоны Telegram/OpenClaw уже подготовлены локально; создание бота, установка Gateway и аудит выполняются на целевом Mac по [runbook Telegram/OpenClaw](openclaw-telegram.md).

Текущее состояние репозитория, стоп-условия и порядок действий для новой сессии зафиксированы в [handoff развёртывания](handoffs/2026-07-16-mac-mini-deployment.md).

## Целевая схема

- OpenClaw — отдельный непривилегированный пользователь macOS;
- Gateway — только loopback;
- Telegram — long polling, без проброса портов роутера;
- Vault Worker — отдельный процесс или ограниченный контейнер;
- vault — отдельный каталог с минимальными правами;
- секреты — вне Git и вне vault;
- резервная копия — отдельна от синхронизации.

## Предварительный аудит

Зафиксировать в приватном эксплуатационном журнале:

- модель Mac mini, процессор, память и свободный диск;
- версию macOS и параметры автоматического обновления;
- состояние FileVault;
- существующие Docker-сервисы и занятые порты;
- способ удалённого доступа;
- текущее резервное копирование;
- поведение после отключения питания;
- режим сна и вход пользователя после перезагрузки.

Не публиковать адрес, имя домашней сети, серийный номер и абсолютные пользовательские пути.

## Подготовка пользователя и каталогов

1. Создать отдельного пользователя без прав администратора.
2. Создать runtime отдельно от vault, выдать доступ только сервисному пользователю и настроить `OBSIDIAN_RUNTIME_PATH`.
3. Выдать сервисному пользователю доступ только к каталогу приложения, runtime и нужной части vault.
4. Не давать доступ к основному Keychain, SSH-ключам и другим домашним сервисам.
5. Хранить `.env` с правами только владельца.
6. Настроить отдельные журналы с ротацией и без текста заметок.

Runtime не является частью vault: в нём временно хранится полный текст необработанных и карантинных событий. Его нельзя помещать внутрь синхронизируемого vault, публиковать или использовать совместно с другим экземпляром worker.

## Синхронизация Obsidian

Если Mac mini используется только как сервер, можно выбрать Obsidian Headless Sync. Если на нём интерактивно работает Desktop Obsidian, использовать Desktop Sync. На одном устройстве нельзя одновременно запускать оба способа для одного vault.

До подключения Sync:

1. создать резервную копию;
2. убедиться, что vault не синхронизируется конкурирующим облачным сервисом;
3. выбрать один механизм;
4. сначала выполнить тест на копии;
5. проверить конфликт одновременного редактирования.

## Перенос репозитория

1. Клонировать репозиторий с закреплённого release или commit.
2. Создать раздельные OpenClaw и worker env-файлы вручную, не копируя секреты через Git.
3. Оставить `OBSIDIAN_DRY_RUN=true`.
4. Запустить `make check` и `doctor`.
5. Подключить тестовый vault и выполнить smoke test.
6. Подключить реальный vault сначала в dry-run.
7. Проверить резервную копию.
8. Только после просмотра отчёта разрешить запись в `00 Inbox`.

## Автозапуск

OpenClaw официально поддерживает установку Gateway как macOS LaunchAgent. LaunchAgent запускается в пользовательской сессии, поэтому после полного отключения питания FileVault или вход пользователя могут потребовать ручного действия.

Перед включением автозапуска проверить:

- отсутствие системного сна;
- перезапуск процесса после сбоя;
- задержку старта до доступности диска и сети;
- health check;
- уведомление, если ежедневный обзор не сформирован;
- контролируемое обновление, а не автоматический переход на неизвестную версию.

## Переключение

В течение недели сохранить старый способ capture и новый сервис параллельно. Сравнивать входящие по `request_id`, но не разрешать двум worker одновременно писать один и тот же материал.

Условия переключения:

- нет потерянных входящих;
- нет перезаписанных заметок;
- backup восстановлен в тестовую папку;
- Telegram allowlist проверен с постороннего ID;
- Gateway недоступен извне;
- ежедневный обзор успешно прошёл минимум три раза.

## Откат

1. Остановить OpenClaw и Vault Worker.
2. Вернуть Telegram в режим только уведомлений или отключить токен.
3. Сохранить карантин и журналы.
4. Восстановить повреждённые файлы из проверенной копии.
5. Не удалять следы инцидента до анализа причины.
