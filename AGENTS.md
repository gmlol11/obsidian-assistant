# Правила работы с репозиторием Obsidian Assistant

## Назначение

Этот репозиторий содержит код и документацию локального ассистента для Obsidian. Относись к проекту как к долгоживущей системе: сохраняй причины решений, поддерживай совместимость и не смешивай код с пользовательскими данными.

## Границы безопасности

- Не добавляй в Git `.env`, API-ключи, токены Telegram, пароли Obsidian, содержимое реального vault, логи с заметками или резервные копии.
- Автоматические тесты могут работать только с `tests/fixtures/vault` или временными каталогами.
- Не направляй тесты и демонстрационные команды на реальное Obsidian-хранилище.
- Сохраняй `dry-run` значением по умолчанию.
- Новые операции записи должны проходить через единый слой политики путей и иметь тесты на выход за пределы vault.
- Не разрешай перезапись, удаление, перемещение или массовое изменение заметок без отдельного решения, подтверждения пользователя и тестов восстановления.
- Сетевые компоненты не должны получать неограниченный доступ к файловой системе.
- OpenClaw должен работать с минимальным набором инструментов, allowlist отправителей и локальным Gateway.
- В проекте не должно быть конфиденциальной информации MOEX. Разрешены только безопасные планы, задачи, напоминания и обезличенные мысли.

## Изменения архитектуры

- Существенные решения фиксируй новым ADR в `docs/decisions`.
- Не переписывай принятый ADR задним числом. Для изменения решения создай новый ADR и пометь прежний как заменённый.
- При изменении поведения обновляй README, соответствующее руководство и секцию `[Unreleased]` в `CHANGELOG.md`.
- Не заявляй интеграцию готовой, пока она не имеет рабочего пути запуска, тестов и инструкции эксплуатации.

## Код

- Поддерживаемая версия — Python 3.12+.
- Используй стандартную библиотеку, если внешняя зависимость не даёт явного выигрыша.
- Проверяй входные данные на границе системы.
- Не выводи секреты или полный текст пользовательских заметок в журналы.
- Новые файлы создавай эксклюзивно; существующие файлы не перезаписывай молча.
- Пути храни в конфигурации, а не в коде.

## Проверка результата

После изменений запусти:

```bash
make check
```

Также проверь:

- `git diff --check`;
- отсутствие секретов и реальных заметок;
- актуальность документации и CHANGELOG;
- соответствие новых возможностей дорожной карте и модели угроз.
